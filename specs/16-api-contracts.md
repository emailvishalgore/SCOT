## 1. Introduction & API Architecture

The SCOT Community Operations Platform operates on a serverless Backend-as-a-Service (BaaS) architecture using **Supabase** (PostgreSQL, Supabase Auth, and Edge Functions) and **Google Drive API** for external document uploads. 

The API layer is split into three communication mechanisms:
1. **PostgREST Auto-generated CRUD:** Default endpoints generated by Supabase from tables (e.g., querying tables, updating simple profiles). Secured via RLS.
2. **Database Stored Procedures (RPCs):** Invoked via `/rest/v1/rpc/<function_name>`. These are used for transaction-heavy, database-bound operations (e.g., recording payments, submitting expenses) where RLS and stored database logic execute atomically.
3. **Supabase Edge Functions:** Hosted Deno-based microservices invoked via `/functions/v1/<endpoint_name>`. These are used for multi-system coordination (e.g., calling the Google Drive API, registering residents while verifying eligibility, and processing fixtures with complex scoring calculations).

This document defines the strict API contracts for the four main business logic gateways in the system:
* `/functions/v1/register-resident` (Event & Sub-Event Registration)
* `/rest/v1/rpc/record_payment` (Override Flat Payment status)
* `/functions/v1/record-score` (Fixture Score Entry and Tie Resolution)
* `/rest/v1/rpc/submit_expense_for_approval` (Expense Approval threshold checks and routing)

### Standard Error JSON Schema
All custom endpoints and RPCs must return error responses in a unified JSON format matching the Supabase standard:
```json
{
  "code": "ERROR_CODE_STRING",
  "message": "Human-readable description of the error.",
  "details": "Technical detail object or message, or null."
}
```

## 2. Event & Sub-Event Registration

### 2.1 Route and Overview
* **Endpoint:** `/functions/v1/register-resident`
* **Method:** `POST`
* **Protocol:** HTTPS / Supabase Edge Function (Deno)
* **Auth Requirement:** JWT with valid session token (`Authorization: Bearer <token>`).

### 2.2 Inputs (Request Payload)
The request body must be a JSON object conforming to the following structure:

```json
{
  "resident_id": "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d",
  "event_id": "1a2beb3c-4d5e-6f7a-8b9c-0d1e2f3a4b5c",
  "sub_event_id": null,
  "registration_method": "SELF",
  "season_id": "c3a8e7e1-88fc-4c4f-9e79-bc6b5a371c26"
}
```

#### Field Specifications & Validation Constraints:
| Field | Type | Required | Constraints / Description |
| :--- | :--- | :--- | :--- |
| `resident_id` | UUID (string) | Yes | Must exist in `core.resident` with `status = 'ACTIVE'`. |
| `event_id` | UUID (string) | Conditional | Must exist in `core.event`. Mandatory if `sub_event_id` is null. |
| `sub_event_id` | UUID (string) | Conditional | Must exist in `core.sub_event`. Mandatory if `event_id` is null. |
| `registration_method`| String ENUM | Yes | One of: `SELF`, `WING_CAPTAIN`, `ON_SPOT`. |
| `season_id` | UUID (string) | Yes | Must match active season. Used to verify flat seasonal dues status. |

### 2.3 Outputs (Response Payloads)

#### Success Response (201 Created)
Returns the created registration record.
```json
{
  "id": "e2d4d5d6-e7e8-4f9a-a0a1-b2b3c4c5d6d7",
  "event_id": "1a2beb3c-4d5e-6f7a-8b9c-0d1e2f3a4b5c",
  "sub_event_id": null,
  "resident_id": "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d",
  "registration_method": "SELF",
  "registered_by_id": "d1c2b3a4-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
  "registered_at": "2026-06-22T20:10:00Z"
}
```

### 2.4 Validation & Business Rules
The Edge Function executes the following validation checks in order:
1. **Target Mutually Exclusive Check:** The payload must contain either `event_id` or `sub_event_id`, but not both. If both or neither are provided, reject with `400 Bad Request`.
2. **Active Target Check:** The target event or sub-event must exist, belong to the provided `season_id`, and have a status of `PLANNED` or `ACTIVE`. If status is `COMPLETED` or `CANCELLED`, reject with `400 Bad Request`.
3. **Resident Status Check:** Verify that the `resident_id` exists in `core.resident` and has a status of `'ACTIVE'`. If not found or inactive, reject with `404 Not Found` or `422 Unprocessable Entity`.
4. **Flat Assignment Check:** Retrieve the flat assigned to the resident for this season from `core.resident_flat_assignment`. If no assignment exists, reject with `422 Unprocessable Entity`.
5. **Dues Payment Check (Finance Gatekeeper):** Invoke `finance.is_flat_eligible(flat_id, season_id)` stored procedure. If the result is `false`, registration is blocked. Reject with `402 Payment Required` or `422 Unprocessable Entity`.
6. **Authorization & Role Verification:** Extract the caller's JWT user metadata claims (`role`, `wing_id`, `flat_id`, `resident_id`):
   - **`HOME_CHIEF` or `HOME_MEMBER` (Resident):** Can only register residents belonging to their own flat. The flat associated with `resident_id` must match the caller's `flat_id` claim. The `registration_method` must be `'SELF'`.
   - **`WING_CAPTAIN` or `WING_COMMANDER`:** Can only register residents of flats belonging to their wing. The wing of the target resident's flat must match the caller's `wing_id` claim. The `registration_method` must be `'WING_CAPTAIN'` or `'SELF'`.
   - **`EVENT_CHAMPION`:** Can register any resident, but only for the events/sub-events they are assigned to in `core.event_assignment`.
   - **`CORE_TEAM` or `SCOT_ADMIN`:** Can register any resident for any event with any method.
   *If authorization fails, reject with `403 Forbidden`.*
7. **Duplicate Check:** Query `core.registration` to ensure the resident is not already registered. If a registration exists, reject with `409 Conflict`.

### 2.5 Error Messages
| HTTP Status | Error Code | Message | Details / Trigger |
| :--- | :--- | :--- | :--- |
| `400 Bad Request` | `INVALID_REGISTRATION_TARGET` | "Must register for either an event or a sub-event, not both." | Either both are provided, or both are null. |
| `400 Bad Request` | `REGISTRATION_CLOSED` | "Cannot register for an event that is already completed or cancelled." | Event status is `COMPLETED` or `CANCELLED`. |
| `401 Unauthorized` | `AUTHENTICATION_REQUIRED` | "Invalid or missing credentials." | Missing or expired JWT token. |
| `403 Forbidden` | `INSUFFICIENT_PRIVILEGES` | "You do not have permission to register this resident." | Resident attempting to register someone outside flat, or Wing Captain outside wing. |
| `404 Not Found` | `RESIDENT_NOT_FOUND` | "The requested resident profile does not exist." | No record matching `resident_id` in database. |
| `409 Conflict` | `DUPLICATE_REGISTRATION` | "Resident is already registered for this event." | Violated `unique_event_registration` constraint. |
| `422 Unprocessable` | `FLAT_ASSIGNMENT_MISSING` | "Resident is not assigned to any flat for this season."| No row in `core.resident_flat_assignment` for `season_id`. |
| `422 Unprocessable` | `FINANCIAL_DUES_PENDING` | "Registration is locked. Annual contribution is pending for Flat {number}." | `finance.is_flat_eligible` returned false. |

## 3. Flat Contribution Payment Override (RPC)

### 3.1 Route and Overview
* **Endpoint:** `/rest/v1/rpc/record_payment`
* **Method:** `POST`
* **Protocol:** Database Stored Procedure (RPC) via PostgREST
* **Auth Requirement:** JWT with valid session token (`Authorization: Bearer <token>`). Allowed Roles: `SCOT_ADMIN`, `CORE_TEAM`, `WING_COMMANDER`.

### 3.2 Inputs (Request Payload)
```json
{
  "target_flat_id": "7a3ce7e1-88fc-4c4f-9e79-bc6b5a371c26",
  "active_season_id": "c3a8e7e1-88fc-4c4f-9e79-bc6b5a371c26",
  "payment_amount": 3000.00,
  "recorder_member_id": "d1c2b3a4-e5f6-7a8b-9c0d-1e2f3a4b5c6d"
}
```

#### Field Specifications & Validation Constraints:
| Field | Type | Required | Constraints / Description |
| :--- | :--- | :--- | :--- |
| `target_flat_id` | UUID (string) | Yes | Must exist in `core.flat`. |
| `active_season_id`| UUID (string) | Yes | Must exist in `core.season` with status `'ACTIVE'`. |
| `payment_amount` | Numeric (2 dec) | Yes | Must be greater than 0. Typically matches default season dues (₹3000.00). |
| `recorder_member_id`| UUID (string) | Yes | Must match the active member recording the payment. |

### 3.3 Outputs (Response Payloads)

#### Success Response (200 OK)
Returns the created or updated `finance.flat_contribution` record.
```json
{
  "id": "f5e6d7c8-b9a0-4123-bcde-f123456789ab",
  "flat_id": "7a3ce7e1-88fc-4c4f-9e79-bc6b5a371c26",
  "season_id": "c3a8e7e1-88fc-4c4f-9e79-bc6b5a371c26",
  "amount": 3000.00,
  "status": "PAID",
  "payment_date": "2026-06-22T20:15:30Z",
  "recorded_by_id": "d1c2b3a4-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
  "receipt_url": null
}
```

### 3.4 Validation & Business Rules
The database stored procedure executes the following security checks and validations:
1. **JWT Claims Verification:** The database function inspects `auth.jwt()`. The user's role claim must be one of `SCOT_ADMIN`, `CORE_TEAM`, or `WING_COMMANDER`. If not, raise an exception.
2. **Wing Boundary Check:** If the caller's role is `WING_COMMANDER`, the flat linked to `target_flat_id` must reside in the wing matching the caller's `wing_id` claim in the JWT. If they attempt to override payment for a flat in another wing, raise an exception.
3. **Season Validity Check:** The `active_season_id` must match a season whose status is `'ACTIVE'`. Payments cannot be recorded for archived seasons.
4. **Member Validation Check:** The `recorder_member_id` must exist as a valid record in `core.member` and associate with the authenticated user's ID.
5. **Database UPSERT:** 
   - Insert a record into `finance.flat_contribution`.
   - On conflict (`flat_id`, `season_id`), update the status to `'PAID'`, update the `payment_date` to `NOW()`, and set the `recorded_by_id`.
6. **Audit Logging:** An database trigger inserts a log entry into `finance.audit_log` capturing the action type (`INSERT` or `UPDATE`), the editor's UUID, the target table, and a JSON representation of the old and new record values.

### 3.5 Error Messages
| HTTP Status | Error Code | Message | Details / Trigger |
| :--- | :--- | :--- | :--- |
| `400 Bad Request` | `INVALID_AMOUNT` | "Payment amount must be greater than zero." | Negative or zero amount provided. |
| `401 Unauthorized` | `AUTHENTICATION_REQUIRED` | "Invalid or missing credentials." | Missing or invalid JWT token. |
| `403 Forbidden` | `P3001` (Supabase generic) | "Unauthorized: Insufficient privileges." | Role is not allowed, or Wing Commander trying to edit a flat outside their wing. |
| `404 Not Found` | `FLAT_NOT_FOUND` | "The specified flat does not exist." | Invalid `target_flat_id`. |
| `422 Unprocessable` | `SEASON_ARCHIVED` | "Cannot record payment for an archived season." | The specified season status is not `'ACTIVE'`. |

## 4. Fixture Score Entry & Leaderboard Recalculation

### 4.1 Route and Overview
* **Endpoint:** `/functions/v1/record-score`
* **Method:** `POST`
* **Protocol:** HTTPS / Supabase Edge Function (Deno)
* **Auth Requirement:** JWT with valid session token (`Authorization: Bearer <token>`). Allowed Roles: `SCOT_ADMIN`, `CORE_TEAM`, or `EVENT_CHAMPION` assigned to the parent event.

### 4.2 Inputs (Request Payload)
```json
{
  "fixture_id": "a1b2c3d4-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
  "scores": [
    {
      "participant_id": "f5e6d7c8-b9a0-4123-bcde-f123456789ab",
      "score": 15.50,
      "placement": 1,
      "attendance_status": "PRESENT"
    },
    {
      "participant_id": "e4d3c2b1-a0f9-4876-bcde-f123456789ab",
      "score": 12.00,
      "placement": 2,
      "attendance_status": "PRESENT"
    }
  ],
  "is_walkover": false,
  "walkover_absent_participant_id": null
}
```

#### Field Specifications & Validation Constraints:
| Field | Type | Required | Constraints / Description |
| :--- | :--- | :--- | :--- |
| `fixture_id` | UUID (string) | Yes | Must exist in `core.fixture`. |
| `scores` | Array | Yes | Must contain score records for all participants assigned to this fixture. |
| `scores[].participant_id` | UUID (string) | Yes | Must match `core.competition_participant.id` for this fixture. |
| `scores[].score` | Numeric (5,2) | No | Raw score achieved. Cannot be negative. |
| `scores[].placement` | Integer | No | Calculated placement rank (1, 2, or 3). |
| `scores[].attendance_status` | String ENUM | Yes | One of: `PRESENT`, `ABSENT`. |
| `is_walkover` | Boolean | Yes | Flag indicating an opponent forfeiture. |
| `walkover_absent_participant_id` | UUID (string) | No | The participant ID of the absent competitor (required if `is_walkover` is true). |

### 4.3 Outputs (Response Payloads)

#### Success Response (200 OK)
Returns the fixture results summary and confirmation of standings update.
```json
{
  "status": "COMPLETED",
  "fixture_id": "a1b2c3d4-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
  "competition_id": "7b8a9c0d-1e2f-3a4b-5c6d-7e8f9a0b1c2d",
  "is_walkover": false,
  "results": [
    {
      "participant_id": "f5e6d7c8-b9a0-4123-bcde-f123456789ab",
      "score": 15.50,
      "placement": 1,
      "attendance_status": "PRESENT",
      "points_awarded": 10.00
    },
    {
      "participant_id": "e4d3c2b1-a0f9-4876-bcde-f123456789ab",
      "score": 12.00,
      "placement": 2,
      "attendance_status": "PRESENT",
      "points_awarded": 7.00
    }
  ],
  "wing_points_updated": [
    {
      "wing_id": "e1d2c3b4-a5f6-7a8b-9c0d-1e2f3a4b5c6d",
      "points_added": 10.00,
      "reason": "1st place: Chess - Fixture 3"
    },
    {
      "wing_id": "b4a3c2d1-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
      "points_added": 7.00,
      "reason": "2nd place: Chess - Fixture 3"
    }
  ]
}
```

### 4.4 Validation & Business Rules
The score recording Edge Function implements the following validation and recalculation lifecycle:
1. **Privilege Verification:** Verify the caller is authorized. If the user's role is `EVENT_CHAMPION`, they must be assigned to the specific event/sub-event of the fixture. `SCOT_ADMIN` and `CORE_TEAM` bypass this check. If unauthorized, reject with `403 Forbidden`.
2. **Fixture State Check:** Ensure the fixture exists and is not already locked. If status is `COMPLETED` and the caller is not `SCOT_ADMIN`, block modifications and return `409 Conflict`.
3. **Walkover Flow Execution:** If `is_walkover` is true:
   - Identify the absent participant via `walkover_absent_participant_id` and ensure they are part of the fixture.
   - Force the absent participant's status to `ABSENT`, score to `0.00`, and placement to `null`.
   - Retrieve competition settings (`walkoverRules`) from `competition.scoring_rule_json`.
   - Set the present participant's status to `PRESENT`. Award points or default score as specified in `walkoverRules.pointsAwardedToPresent` (e.g. 2 points) and `walkoverRules.defaultScore` (e.g. score of 2.00).
   - If `walkoverRules.forfeitParticipationPointsOnWalkover` is true, the absent participant receives 0 participation points.
4. **Standard Score Flow Execution:** If `is_walkover` is false:
   - Match all entries in `scores` against the active participants in `core.competition_participant` for this fixture.
   - Reject with `400 Bad Request` if any participant is missing or if extraneous participant IDs are submitted.
   - Enforce non-negative score inputs.
5. **Tie Resolution Algorithm:** If raw scores or placements are tied, fetch `tiedPlacementResolution` from the competition settings:
   - **`SPLIT`:** For two participants tied for 1st place, calculate points by adding 1st and 2nd place configurations and dividing by two:
     $$\text{Points} = \frac{P_{first} + P_{second}}{2}$$
     No 2nd place is awarded; the next participant receives 3rd.
   - **`FULL`:** Both participants are awarded full 1st place points.
6. **Leaderboard Standings Recalculation:**
   - Update `attendance_status`, `score`, and `placement` for participants in `core.competition_participant`.
   - Transition `fixture.status` to `COMPLETED`.
   - Retrieve the competition type (`INDIVIDUAL` vs `WING_BASED`) and season ID.
   - **Wing Standings Aggregation:**
     - For `WING_BASED` competitions, write the points directly to the wing's record.
     - For `INDIVIDUAL` competitions, query `allowMultiplePlacementsPerWing` and `participationPointsCap` ($C_{part}$) from `scoringRuleJson`:
       - If multiple placements are allowed, sum all individual resident points for the wing.
       - If multiple placements are disabled, select only the highest scoring resident's placement points for the wing, and add participation points (default ₹2.00) for other residents up to $C_{part}$ cap.
     - Insert or update the records in `core.wing_score` tracking the season, competition, points awarded, and reason.

### 4.5 Error Messages
| HTTP Status | Error Code | Message | Details / Trigger |
| :--- | :--- | :--- | :--- |
| `400 Bad Request` | `PARTICIPANT_MISMATCH` | "Scores array does not match the participants assigned to this fixture." | Wrong participant IDs or count. |
| `400 Bad Request` | `INVALID_WALKOVER_PARTICIPANT`| "The walkover participant is not part of this fixture." | Absent ID not belonging to fixture. |
| `401 Unauthorized` | `AUTHENTICATION_REQUIRED` | "Invalid or missing credentials." | Missing or expired JWT. |
| `403 Forbidden` | `INSUFFICIENT_PRIVILEGES` | "You are not assigned as the Champion for this event." | Event Champion trying to edit unassigned event fixture. |
| `404 Not Found` | `FIXTURE_NOT_FOUND` | "The requested fixture does not exist." | Invalid `fixture_id`. |
| `409 Conflict` | `FIXTURE_LOCKED` | "Fixture results are already finalized." | Fixture is `COMPLETED` (non-admin caller). |

## 5. Expense Approval Routing & Submission (RPC)

### 5.1 Route and Overview
* **Endpoint:** `/rest/v1/rpc/submit_expense_for_approval`
* **Method:** `POST`
* **Protocol:** Database Stored Procedure (RPC) via PostgREST
* **Auth Requirement:** JWT with valid session token (`Authorization: Bearer <token>`). Allowed Roles: `EVENT_CHAMPION`, `CORE_TEAM`, `SCOT_ADMIN`.

### 5.2 Inputs (Request Payload)
```json
{
  "target_expense_id": "a9b8c7d6-e5f6-7a8b-9c0d-1e2f3a4b5c6d"
}
```

#### Field Specifications & Validation Constraints:
| Field | Type | Required | Constraints / Description |
| :--- | :--- | :--- | :--- |
| `target_expense_id` | UUID (string) | Yes | Must exist in `finance.expense` in `'DRAFT'` status. |

### 5.3 Outputs (Response Payloads)

#### Success Response (200 OK)
Returns the updated `finance.expense` record with its newly routed status.
```json
{
  "id": "a9b8c7d6-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
  "season_id": "c3a8e7e1-88fc-4c4f-9e79-bc6b5a371c26",
  "category": "PRIZES",
  "vendor_id": null,
  "event_id": "1a2beb3c-4d5e-6f7a-8b9c-0d1e2f3a4b5c",
  "description": "Trophies for Cricket Tournament",
  "amount": 4500.00,
  "receipt_url": "https://drive.google.com/open?id=1xyz...",
  "status": "PENDING_APPROVAL",
  "approved_by_id": null
}
```

### 5.4 Validation & Business Rules
The database stored procedure executes the following logic:
1. **JWT Verification:** Validate that the caller has a valid role claim (`EVENT_CHAMPION`, `CORE_TEAM`, or `SCOT_ADMIN`) in their JWT metadata. If not, raise an exception.
2. **Ownership Check:** If the caller's role is `EVENT_CHAMPION`:
   - Verify they are either the creator of this expense, or they are assigned to the event linked to the expense (matched using `event_id` in `core.event_assignment`). If not, reject.
3. **Status Check:** Validate that the target expense is currently in `'DRAFT'` status. If the status is already `PENDING_APPROVAL`, `APPROVED`, or `DISBURSED`, raise an exception to prevent duplicate submissions.
4. **Configuration Lookup:** Read the active season's `financeApprovalConfig` from the `core.system_config` table (keyed by `financeApprovalConfig` for the active season).
   - If no config is found, fall back to standard defaults:
     - `autoApprovalLimit`: 500.00
     - `singleOwnerApprovalLimit`: 5000.00
     - `dualOwnerApprovalLimit`: 20000.00
5. **Threshold Routing Evaluation:**
   - **Auto-Approval (Amount $\le$ `autoApprovalLimit`):**
     - Update status directly to `'APPROVED'`.
     - Set `approved_by_id` to `NULL` (indicating system auto-approval).
   - **Needs Review (Amount > `autoApprovalLimit`):**
     - Update status to `'PENDING_APPROVAL'`.
     - The expense is routed for offline/online review. The approval workflows evaluate additional limits during final signature:
       - **Single Owner Approval:** If amount $\le$ `singleOwnerApprovalLimit`, the expense can be finalized by a single assigned Event Champion or Portfolio Owner calling `finance.approve_expense`.
       - **Dual Core Team Approval:** If `singleOwnerApprovalLimit` < amount $\le$ `dualOwnerApprovalLimit`, the status remains pending until two unique members of the `CORE_TEAM` approve the record.
       - **General Body Approval:** If amount > `dualOwnerApprovalLimit`, approval requires `SCOT_ADMIN` role execution and verification that a general assembly minutes document link exists in `receipt_url` or transaction notes.
6. **Audit Trail:** Write a record into `finance.audit_log` detailing the state transition.

### 5.5 Error Messages
| HTTP Status | Error Code | Message | Details / Trigger |
| :--- | :--- | :--- | :--- |
| `401 Unauthorized` | `AUTHENTICATION_REQUIRED` | "Invalid or missing credentials." | Missing or invalid JWT. |
| `403 Forbidden` | `P3001` (Supabase generic) | "Unauthorized: Insufficient privileges." | Caller is not authorized to submit this expense. |
| `404 Not Found` | `EXPENSE_NOT_FOUND` | "The requested expense draft does not exist." | Invalid `target_expense_id`. |
| `409 Conflict` | `INVALID_EXPENSE_STATE` | "Only draft expenses can be submitted for approval."| Status is already `PENDING_APPROVAL`, `APPROVED`, etc. |
| `500 Server Error` | `CONFIG_NOT_FOUND` | "System configuration for finance approvals is missing."| Key `financeApprovalConfig` not found in `core.system_config`. |
